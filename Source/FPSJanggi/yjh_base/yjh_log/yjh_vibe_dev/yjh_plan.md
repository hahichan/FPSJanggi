# YJH ArenaBattleReal 바이브 코딩 계획서 (초안)

## 0. 문서 목적
- 본 문서는 바이브 코딩 방식으로 1대1 전투 시스템을 빠르게 구현하되, 구조 붕괴 없이 유지보수 가능한 형태를 유지하기 위한 실행 초안이다.
- 핵심 원칙은 계획서 분리와 컴포지션 우선 설계다.
- 구현 전 기준: 작업 단위, 완료 기준, 검증 기준, 롤백 기준을 먼저 고정한다.

## 1. 계획서 분리 원칙
본 문서는 총괄 계획서이며, 실제 구현 전에 아래 두 문서로 분리해 동시 운영한다.

1. 게임 흐름 계획서(모드/오케스트레이션)
- 소유 책임: 전투 진입, 승패 확정, 결과 반영, 보드 복귀
- 비책임: 캐릭터 내부 이동/피해 계산 상세

2. 베이스 캐릭터 계획서(전투 단위)
- 소유 책임: 이동, 시점, 입력, HP, 피격/사망 이벤트
- 비책임: 매치 전체 승패 오케스트레이션

3. 통합 접점 시트(얇은 문서)
- 서로 호출하는 함수와 이벤트 이름만 관리
- 중복 설명 금지, 계약만 기록

## 2. 아키텍처 방향: 다중 상속보다 컴포지션
1. 기본 방침
- 기능 구현은 컴포넌트 조합 중심으로 설계
- 다중 상속은 인터페이스 구현 용도로만 제한

2. 이유
- 책임 분리가 명확해 유지보수 용이
- 기능 교체와 테스트가 쉬움
- 서버 권한 로직을 기능별로 분리해 멀티플레이 안정성 확보

3. 목표 구조
- yjh_ArenaCombatantBase: 이동/시점/입력 뼈대
- yjh_ArenaHealthComponent: 체력/피해/사망 처리
- yjh_ArenaCombatComponent: 공격 요청/검증/재사용 가능한 전투 훅
- yjh_ArenaSubStateMachine: ArenaBattleReal 서브 상태 전환

## 3. 현재 기준선
- NSH가 메인 페이즈를 소유
- YJH 진입 경로(UI/F8/콘솔) 및 스텁 동작은 확보됨
- 지금부터는 "실전투용 베이스 캐릭터 + 룰 연결"로 확장

## 4. 최종 목표(MVP)
1. ArenaBattleReal에서 서버 권한 1대1 전투 진행
2. yjh_ 베이스 캐릭터 기반 양측 전투 유닛 구동
3. HP 0 기준 사망/패배 1회 확정
4. 결과를 NSH 보드 흐름에 중복 없이 반영

## 5. 비범위(이번 사이클 제외)
- 맵 트래블 구조 개편
- NSH 메인 enum 대규모 개편
- 연출/사운드 폴리싱
- 밸런스 고도화

## 6. 바이브 코딩 작업 단위

### Step A: 계약 먼저 고정
- 게임 흐름 계획서에 "전투 시작/종료 계약" 정의
- 베이스 캐릭터 계획서에 "캐릭터 책임" 정의
- 통합 접점 시트에 함수명/RPC/Event 표준명 확정

완료 기준
- 구현 없이도 시작-전투-종료 시퀀스를 텍스트로 추적 가능

### Step B: 캐릭터 뼈대
- yjh_ArenaCombatantBase 생성
- 이동/시점/입력 및 전투 컴포넌트 부착 지점만 준비

완료 기준
- 전투 캐릭터 스폰 및 기본 조작 가능

### Step C: 체력/사망 루프
- yjh_ArenaHealthComponent 생성
- 서버 권한 HP 감소/사망 이벤트/복제 동기화 준비

완료 기준
- 서버에서 피해 적용 시 클라에 동일 체력 반영

### Step D: 전투 룰 최소 루프
- yjh_ArenaCombatComponent에서 테스트 공격 루프 연결
- 사망 시 패배 보고 1회 보장

완료 기준
- HP 0에서 승패 확정 이벤트가 단 1회 발생

### Step E: NSH 접점 연결
- 기존 NSH 흐름에 결과 반영 훅만 최소 연결

완료 기준
- 전투 종료 후 보드 복귀가 중복 없이 수행

## 7. 네트워크/안정성 필수 체크
- 서버만 승패 확정
- 패배 보고 중복 방지 플래그 필수
- RPC 연타 제한
- 전투 중 이탈/재접속 시 기본 처리 정책 기록

## 8. 검증 시나리오(매 Step 공통)
1. Host + Client 접속
2. YJH Arena Start 진입
3. 상태 전환 동기화 확인
4. 공격-피해-사망-승패 순서 확인
5. 보드 복귀 확인
6. 3~5회 반복 안정성 확인

## 9. 롤백 기준
- Step 완료 기준 미달 시 다음 Step 진행 금지
- 빌드 실패/동기화 불일치 발생 시 직전 안정 커밋 기준으로 되돌릴 수 있게 변경 단위 최소화

## 10. 운영 규칙
- 신규 파일은 yjh_ 접두어 사용
- 새로운 코드 작성/의미 있는 변경마다 ../yjh_ai_logs/yjh_chat_2.md 기록
- 큰 변경 1회보다 작은 변경 다회 + 즉시 검증

## 12. 확장성 최우선 원칙(중요)
- 베이스 캐릭터는 "작성자 외 개발자"가 파생 캐릭터를 쉽게 만들 수 있어야 한다.
- 파생 캐릭터 제작자는 가능한 한 C++ 수정 없이 Blueprint/DataAsset 중심으로 작업 가능해야 한다.
- Base 클래스에는 특정 캐릭터 전용 하드코딩을 금지하고, 확장 포인트를 명시적으로 제공한다.
