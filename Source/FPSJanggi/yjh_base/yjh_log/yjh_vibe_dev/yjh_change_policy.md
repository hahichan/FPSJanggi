# YJH 변경 규칙 정책서 (v0.1)

## 0. 목적
- 대화 기반 개발 중 구조 붕괴를 막기 위해 변경 허용 범위와 심사 기준을 고정한다.
- 빠른 실험은 유지하되, 파괴적 변경은 통제한다.

## 1. 변경 등급
### A등급: 자유 변경(즉시 반영 가능)
- 신규 함수/이벤트 추가
- 로그/디버그 정보 추가
- DataAsset 키 신규 추가(기존 키 유지 조건)
- Blueprint 확장 포인트 추가

### B등급: 조건부 변경(문서 동기화 필요)
- 함수 인자 추가(기존 호출 호환 유지)
- 구조체 필드 추가(기본값 제공)
- 상태 enum 항목 추가(기존 상태 의미 유지)

### C등급: 파괴적 변경(심사 후 반영)
- 공개 함수 시그니처 삭제/이름 변경
- 기존 인자 의미 변경
- 기존 DataAsset 키 삭제/재명명
- 서버 권한 경계 변경

## 2. 파괴적 변경 절차(C등급)
1. 변경 제안서 3줄 작성(배경/영향/대안)
2. 통합 접점 시트 버전 증가
3. 게임모드/베이스캐릭터 계획서 동시 수정
4. 회귀 체크리스트 최소 1회 수행

## 3. 확장성 보호 규칙(핵심)
- 파생 캐릭터 제작자는 가능한 한 C++ 수정 없이 작업 가능해야 한다.
- Base 클래스에 특정 캐릭터 하드코딩 금지.
- 새 캐릭터 추가 시 수정 파일 3개 이하를 목표로 유지.

## 4. 버전 규칙
- 문서 버전: MAJOR.MINOR
- MAJOR: 파괴적 변경 발생
- MINOR: 하위 호환 추가 변경

## 5. 승인/기록 규칙
- 의미 있는 변경은 ../yjh_ai_logs/yjh_chat_2.md에 작업 번호로 기록
- C등급 변경은 반드시 변경 이유와 롤백 기준을 함께 기록

## 6. 계획 단계 가변성 규칙
- 현재 단계는 계획서 중심 작업이므로 문서는 언제든 수정 가능하다.
- 코드 미구현 상태에서는 문서 변경을 실험으로 간주하고 유연하게 반영한다.
- 단, 공통 용어(CombatSessionId/CombatantId/EndReason/FinalizeOnce) 변경 시 관련 문서 동기화는 유지한다.
