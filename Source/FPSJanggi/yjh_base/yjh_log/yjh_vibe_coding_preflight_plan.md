# YJH 바이브코딩 사전 계획서 (Stuck-Guard v1)

## 0. 문서 목적
- 지금 단계는 구현이 아니라 "바이브코딩 전에 길을 막히지 않게 설계하는 단계"로 고정한다.
- 중간에 막혔을 때도 복구 가능한 작업 단위/의사결정 규칙/중단 규칙을 미리 정의한다.

## 1. 현재 선언 (중요)
- 본 사이클은 계획서 작성만 수행한다.
- 코드 작성/리팩터링/블루프린트 수정은 게이트 통과 전 금지한다.

## 2. 바이브코딩 핵심 리스크
1. 문서 없이 바로 구현 시작 -> 중간 분기 폭증
2. 문제 발생 시 되돌아갈 기준 없음
3. 함수/데이터 이름이 세션마다 바뀜
4. 원인 불명 상태에서 패치 누적

## 3. Stuck-Guard 운영 원칙
1. One Goal Per Session
- 세션마다 목표 1개만 허용한다.
- 예: "Slot1~Slot2 요청/검증/쿨타임 루프만 확인"

2. One Decision Per File
- 한 파일에는 한 종류의 결정만 기록한다.
- 스킬데이터 결정과 UI 결정을 같은 파일에 섞지 않는다.

3. Freeze Vocabulary
- 고정 용어를 세션 내 변경 금지한다.
- CombatSessionId, CombatantId, SkillId, SlotIndex, RemainingSec

4. Fail-Fast Logging
- 실패코드는 문자열이 아니라 사전 정의 코드로만 사용한다.
- Validation: SKV_*
- Execution: SKX_*

5. Stop Condition First
- 작업 시작 전에 "중단 조건"을 먼저 기록한다.
- 중단 조건 충족 시 기능 추가 금지, 원인 복구만 수행한다.

## 4. 게이트 기반 진행 (코드 전)

### Gate A: 계약 고정
필수 확인
- 스킬데이터 필드 표가 최신인지
- 실행기 파이프라인이 최신인지
- UI 바인딩 이벤트가 최신인지

통과 기준
- 문서 간 키 이름 불일치 0건

### Gate B: 실패코드 고정
필수 확인
- SKV/SKX 코드가 중복 없이 정의되어 있는지
- 로그 최소 필드가 정의되어 있는지

통과 기준
- 실패 시나리오 5개 이상을 코드가 아닌 텍스트로 재현 가능

### Gate C: 최소 수직 슬라이스 정의
필수 확인
- Slot1~Slot2만 대상으로 입력->검증->실행->쿨타임->UI 갱신 경로를 텍스트로 고정

통과 기준
- 팀원이 문서만 보고 동일 순서로 설명 가능

## 5. 막힘 발생 시 복구 프로토콜 (핵심)

### 5.1 막힘 정의
아래 중 1개라도 해당하면 막힘 상태로 판정
- 30분 이상 동일 오류 반복
- 원인 가설 3개 이상으로 분산
- 같은 파일 재수정 3회 이상

### 5.2 복구 절차
1. 증상 3줄 기록
- 무엇을 하려 했는지
- 실제로 무엇이 깨졌는지
- 마지막 정상 상태가 무엇인지

2. 영향 범위 절단
- 세션 목표 외 영역 수정 금지
- 관련 없는 문서/코드 변경 취소(단, 타인 변경은 보존)

3. 계약 재확인
- yjh_skill_data_format_spec.md
- yjh_skill_executor_spec.md
- yjh_ui_data_binding_contract.md

4. 최소 루프로 축소
- Slot1 단일 스킬만 남기고 설명 가능 상태 복귀

5. 재시작 결정
- 같은 날 재시도 1회만 허용
- 재시도 실패 시 다음 세션으로 이월

## 6. 바이브코딩 프롬프트 템플릿

### 6.1 기능 추가 요청 템플릿
- 목표: (한 줄)
- 범위: (파일 1~3개)
- 비범위: (건드리면 안 되는 것)
- 완료 기준: (관측 가능한 결과 2~3개)
- 중단 조건: (실패 기준)

### 6.2 버그 수정 요청 템플릿
- 증상: (한 줄)
- 재현 순서: (3단계)
- 기대/실제: (각 1줄)
- 의심 구간: (파일 1~2개)
- 금지 변경: (관계 없는 시스템)

## 7. 파일 운영 규칙
1. 계획서 우선 파일
- yjh_skill_data_format_spec.md
- yjh_skill_executor_spec.md
- yjh_skill_initial_catalog.md
- yjh_ui_data_binding_contract.md

2. Delta 파일 규칙

- 파일명: yjh_ui_binding_delta_YYYYMMDD_NN.md

3. 작업 로그 규칙
- 의미 있는 결정은 yjh_chat_2.md에 작업 번호로 기록
