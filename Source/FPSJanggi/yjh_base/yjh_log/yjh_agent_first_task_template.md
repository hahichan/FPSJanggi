## 2. 구현 전 고정 체크
- [ ] 하드코딩 금지: 캐릭터/스킬 전용 분기문을 Base 클래스에 추가하지 않는다.
- [ ] 서버 권한 고정: 최종 판정/종료/쿨타임 확정은 서버에서만 수행한다.
- [ ] 데이터 소스 고정: SkillData는 읽기 전용, 동적값은 RuntimeContext에서만 읽는다.
- [ ] 파이프라인 단계 명시: 입력 -> 검증 -> 실행 -> 반영 -> 이벤트를 코드 구조로 분리한다.
- [ ] 실패 코드 표준화: 실패 사유는 FailCode로만 외부에 노출한다.

## 3. 구현 단계 체크리스트
1. 계약 선언 단계
- [ ] 함수 시그니처/이벤트 이름을 문서와 일치시킨다.
- [ ] 입력/출력 구조체를 먼저 고정하고 구현 시작한다.

2. 파이프라인 구현 단계
- [ ] ReceiveInput 계층과 Execute 계층을 분리한다.
- [ ] Validate 계층을 단일 함수 또는 명시 함수군으로 유지한다.
- [ ] ApplyResult 계층에서만 상태를 변경한다.
- [ ] EmitEvents 계층에서만 UI/로그 이벤트를 발생시킨다.

3. 확장성 점검 단계
- [ ] 캐릭터 추가 시 수정 파일 수를 최소화한다.
- [ ] 새 캐릭터는 DataAsset/BP 조합으로 추가 가능해야 한다.
- [ ] 코드 변경 없이 Slot 매핑/스탯/스킬 파라미터 교체가 가능해야 한다.

## 4. 품질 게이트 (머지 전 필수)
- [ ] 가독성: 함수 길이 과다/중첩 분기 과다를 제거했다.
- [ ] 유지보수성: 하드코딩 상수 대신 설정값/데이터 참조를 사용했다.
- [ ] 재현성: 동일 입력에서 서버 기준 결과가 일관되다.
- [ ] 관측성: 핵심 실패 경로에 로그 키와 최소 필드가 남는다.
- [ ] 안전성: FinalizeOnce, Timeout/Grace, RuntimeContext 규칙 위반이 없다.

## 5. 테스트 최소 세트
- [ ] 정상 종료 3회 연속
- [ ] 동시 종료 후보 경합 3회
- [ ] Timeout + Grace 겹침 3회
- [ ] SkillData 불량(매핑 누락/잘못된 슬롯) 3회
- [ ] 신규 캐릭터 DataAsset 교체 후 회귀 테스트 1회
