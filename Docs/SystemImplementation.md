# FPS 장기 핵심 시스템 구현 상태

## Steam 세션

`USessionSubsystem`이 다음 기능을 제공한다.

- `CreateRoom`: Steam 로비를 생성하고 `MainMap?listen`으로 이동
- `FindRooms`: `FPSJANGGI_GAME=FPSJANGGI_V1` 방 검색
- `JoinRoom`: 검색 결과의 접속 주소로 `ClientTravel`
- `QuickMatch`: 참가 가능한 방에 입장하고 없으면 새 방 생성
- `LeaveRoom`: 현재 세션 종료

개발용 Steam App ID는 480이다. 실제 전송 검사는 서로 다른 Steam 계정과 PC에서 수행해야 한다.

## 프런트엔드 로비

단독 실행 또는 패키지로 `MainMap`을 열면 장기판이나 진형 선택 UI보다 먼저 로비가 표시된다. 언리얼의 `2 Players + Listen Server`는 이미 두 플레이어를 직접 연결한 개발용 회귀 테스트이므로 로비를 건너뛰고 기존 멀티플레이 대국 흐름을 시작한다.

- 왼쪽에는 게임 제목과 설명, 오른쪽에는 대국실 목록을 배치한다.
- 방 이름을 입력하고 `방 만들기`를 누르면 Steam 세션을 만든 뒤 `MainMap?listen`으로 이동한다.
- `새로고침`은 참가 가능한 대국실을 검색하며 각 방 카드에는 방 이름, 호스트, 인원, 핑과 `합류` 버튼이 표시된다.
- `빠른 대전`은 참가 가능한 방에 먼저 합류하고, 방이 없으면 새 방을 만든다.
- 세션 검색·생성·합류 결과는 로비 하단 상태 문구로 표시한다.

### 로비 카메라를 에디터에서 지정하는 방법

로비는 레벨에 저장된 전용 `CameraActor`의 위치, 회전, FOV를 그대로 사용한다. 기존의 두 네모 핸들은 실제 카메라가 아니라 전환 전 초기 위치와 초점이며, 전용 카메라가 없을 때만 임시 카메라를 생성하는 예비값이다.

1. PIE를 종료한다.
2. World Outliner에서 `JanggiBoard1` Actor를 선택한다.
3. Details의 `FPS Janggi > Lobby > Placement`에서 `Create Lobby Camera From Offset Handles`를 누른다.
4. 현재 두 핸들의 위치와 초점으로 `FPSJanggi Lobby Camera`가 생성되고 `Lobby Camera Actor`에 자동 연결되는지 확인한다.
5. 자유 뷰포트를 실제 로비에서 보여 줄 구도로 이동한다. 이때 화면 안에 카메라 아이콘이 보이는 것은 아직 해당 카메라 시점이 아니라는 뜻이다.
6. 다시 `JanggiBoard1`을 선택하고 `Snap Lobby Camera To Current Editor View`를 누른다.
7. `FPSJanggi Lobby Camera`의 Camera Preview 또는 Pilot 기능으로 같은 구도인지 확인한다.
8. `MainMap`을 저장한다.

생성 버튼은 최초 한 번만 카메라를 만들며, 이미 카메라가 있으면 그 위치·회전·FOV를 변경하지 않는다. 지정한 카메라 참조와 트랜스폼은 `MainMap`에 저장되어 팀원에게도 동일하게 전달된다. 생성된 카메라는 `Lobby` 태그도 가지므로 Blueprint 재컴파일로 직접 참조가 일시적으로 유실되어도 로비가 같은 카메라를 찾는다. `Lobby Camera Actor`를 비워 두고 태그 카메라도 없을 때만 기존 오프셋이 대체값으로 사용된다.

## 서버 권한 장기판

`AAuthoritativeJanggiBoard`가 유일한 규칙 판정자다.

- 서버가 `Start_Figure[90]`을 읽어 Blue/Red 기물 Blueprint 32개를 생성한다.
- 클라이언트는 좌표만 RPC로 요청하고 이동·충돌·제거는 서버가 결정한다.
- 졸, 차, 포, 마, 상, 사, 궁의 행마법과 길막을 C++에서 검증한다.
- 턴, 클릭 기록, 제한 시간, 보드 정지, 전투 상태를 복제한다.
- 기본 턴 제한은 60초이며 만료 시 선택을 취소하고 상대 턴으로 넘긴다.
- 같은 기물의 기본 연속 이동 제한은 2회다.

`origin/master`의 원본 `JanggiBoard1` 값을 사용한다.

- 좌표: `X=(index / 9) * 8.0`, `Y=(index % 9) * 8.5`
- 기물 Actor 스케일: `1.8`

## 진형 선택

기존 로컬 전용 `InterFace_MS` 대신 C++ 진형 선택 UI를 사용한다. 양쪽 선택이 서버에 도착할 때까지 보드 입력을 정지하며, 선택이 완료되면 32개 기물을 생성하고 Blue 턴을 시작한다.

장기에서는 시작 전에 각 플레이어가 뒷줄 좌우의 서로 붙어 있는 마와 상 위치를 바꿀 수 있다. 따라서 `마-상-상-마`, `마-상-마-상`, `상-마-상-마`, `상-마-마-상` 네 배치는 정상 규칙이다. 이 선택은 마/상의 순서만 바꾸며 청·한 진영의 보드 끝이나 카메라 방향은 바꾸지 않는다.

## 이동 가능 칸 표시

기물을 선택하면 서버가 현재 보드 점유 상태와 실제 행마법으로 90칸을 검사한다. 합법적인 목적지 인덱스만 해당 플레이어에게 Client RPC로 전송한다.

- 목적지는 장기판 위의 2중 옥색 원형 토큰으로 표시한다. 바깥쪽의 짙은 테두리와 안쪽의 밝은 옥색 면을 겹쳐 기존 형광 초록 구보다 장기판 분위기에 맞추고 교차점을 가리지 않는다.
- 화면 좌표 UMG 투영을 사용하지 않는다. 각 합법 칸의 실제 장기판 로컬 좌표 위에 엔진 기본 원통 두 개를 낮게 배치하므로 카메라 각도와 화면 해상도에 관계없이 교차점 위에서 보인다.
- 마커는 로컬 클라이언트에만 생성하고 충돌을 비활성화하므로 보드 클릭이나 네트워크 복제에 영향을 주지 않는다.
- 화면 재투영 타이머가 없다. 바깥/안쪽 마커 컴포넌트는 필요한 수만큼 최초 한 번만 만들고, 이후 선택에서는 위치와 표시 상태만 바꾸는 풀 방식으로 재사용한다.
- 같은 편의 다른 기물을 선택하면 목록을 즉시 갱신한다.
- 이동, 턴 만료, 턴 전환, 아레나 진입 시 모두 숨긴다.
- 이동·턴 전환 시 화면 마커를 즉시 제거한다.
- 상대 기물, 빈 칸 또는 불가능한 목적지를 누르면 상단에 거절 이유를 1.5초 동안 표시한다.

## 턴 표시 UI

각 로컬 플레이어 화면 상단에 다음 정보를 표시한다.

- `내 차례` 또는 `상대 차례`
- 현재 팀: 청 또는 한
- 남은 턴 시간
- 진형 선택/상대 대기 상태
- 기물 충돌 전투 진행 상태

Blue 턴은 파란 배경, Red 턴은 붉은 배경으로 표시한다. UI는 `HitTestInvisible`이므로 보드 클릭을 방해하지 않는다.

## UI 시각 체계

기존의 단색 네모와 기본 글꼴 조합 대신 코드 UI 전체가 하나의 공통 스타일을 사용한다.

- 먹색 반투명 패널, 상아색 본문, 금색 테두리와 강조색
- 둥근 모서리, Hover/Pressed 상태가 구분되는 버튼
- 청·한 팀 색을 유지하면서 채도를 낮춘 턴 패널
- 제목, 설명, 보조 문구의 크기와 굵기를 분리한 정보 위계
- 진형 선택 버튼에는 배치 이름과 좌우 기물 순서를 함께 표시
- 전투 디버그 버튼도 같은 패널·버튼 체계를 사용하되 청/한 색으로 즉시 구분

공통 색상과 패널·버튼·텍스트 스타일은 `JanggiUIStyle`에 모아 두었으므로 이후 로고 이미지나 전용 폰트를 추가하더라도 한 곳에서 전체 UI를 조정할 수 있다.

## 카메라

MainMap에 원래 배치된 두 카메라를 유지한다.

- Blue: `CameraActor_1`
- Red: `CameraActor_2`
- 배치 카메라의 위치·시야각·바라보는 방향을 기준으로 팀별 로컬 카메라를 만들고, 자기 진영이 항상 화면 아래쪽이 되도록 화면 상하축만 자동 보정한다. 진형 선택 결과와는 무관하다.
- 아레나 진입 직전 각 로컬 플레이어의 실제 ViewTarget을 저장하고, 종료 후 팀 이름을 다시 추측하지 않고 저장한 카메라로 복귀한다.

## 충돌과 아레나 전환

상대 기물이 있는 합법 목적지를 선택하면 보드 입력을 정지하고 `ArenaTransition`으로 전환한다. 전투 시스템은 `ResolveArenaBattle(WinnerTeam)`을 호출해 결과를 서버에 전달한다.

현재 전투 시스템 연동 전 임시 단계에서는 충돌한 기물 종류에 대응하는 스켈레탈 메시 캐릭터 두 명만 생성한다.

- 청/한 임시 캐릭터와 양쪽 카메라 위치를 배치된 `JanggiBoard1` 인스턴스의 Details 패널에서 직접 지정할 수 있다.
- `Arena Blue Spawn Offset`, `Arena Red Spawn Offset`: 두 임시 캐릭터의 위치
- `Arena Blue Camera Offset`, `Arena Red Camera Offset`: 각 팀이 전투 중 사용할 카메라 위치
- `Arena Camera Focus Offset`: 양쪽 전투 카메라가 바라볼 위치
- 다섯 값은 장기판 Actor 기준 상대 좌표이며 `Make Edit Widget`이 적용되어 레벨 뷰포트의 3D 핸들로도 옮길 수 있다.
- 임시 캐릭터의 메시와 위치는 서버에서 생성·복제
- Development 빌드에서만 하단에 `청 승리로 전투 종료`, `한 승리로 전투 종료` 버튼 표시
- 디버그 승리 요청도 서버가 검증하고 `ResolveArenaBattle`을 호출

- 공격자 승리: 수비 기물 제거 후 공격 기물 이동
- 수비자 승리: 공격 기물 제거
- 처리 후 상대 턴 시작 및 기존 보드 카메라 복귀
- 패배한 기물이 궁/왕이면 즉시 게임을 종료하고 승리 팀을 양쪽 화면에 표시하며 보드 입력과 턴 타이머를 정지
- 경기 종료 패널의 `로비로 돌아가기`를 누르면 서버가 원격 클라이언트를 먼저 로비로 보낸 뒤 호스트를 이동시킨다. 활성 Steam 세션이 있으면 세션을 정리하고, 양쪽 모두 독립 로비 화면과 로비 카메라로 복귀한다.

### 전투 위치를 에디터에서 지정하는 방법

1. PIE를 종료한다.
2. World Outliner에서 레벨에 배치된 `JanggiBoard1` Actor를 선택한다.
3. Details에서 `FPS Janggi > Arena > Placement`를 연다.
4. 청/한 스폰, 청/한 카메라, 카메라 초점의 다섯 위치를 숫자로 입력하거나 뷰포트의 위치 핸들을 드래그한다.
5. 플레이해 양쪽 화면을 확인한 뒤 카메라가 캐릭터를 바라보도록 스폰과 초점 위치를 미세 조정한다.

이 설정은 레벨에 저장되므로 팀원이 같은 `MainMap`을 받으면 같은 전투 위치를 사용한다.

## 다른 담당자 작업 연결 규약

팀원에게 전달할 역할별 작업 순서, Blueprint 연결법, 브랜치 규칙과 제출 양식은 [TeamIntegrationGuide.md](TeamIntegrationGuide.md)에 정리되어 있다.

핵심 장기 흐름은 그대로 유지하고 실제 FPS 캐릭터만 교체할 수 있도록 `JanggiBoard1`에 선택형 연결 슬롯을 제공한다. 슬롯이 비어 있으면 현재 임시 캐릭터를 사용하므로 한 담당자의 미완성 작업 때문에 기존 대국이 중단되지 않는다.

### 캐릭터 담당자: 마·상·졸 / 사·차·포 / 궁

1. 전투 캐릭터는 `Pawn` 또는 `Character` 기반 Blueprint로 만든다.
2. PIE를 종료하고 `MainMap`의 `JanggiBoard1`을 선택한다.
3. Details의 `FPS Janggi > Arena > Integration`에서 `Blue Arena Combatant Classes`와 `Red Arena Combatant Classes`를 연다.
4. 각 팀의 `Horse`, `Elephant`, `Soldier`, `Guard`, `Chariot`, `Cannon`, `General` 키에 해당 Pawn Blueprint Class를 지정한다.
5. 지정된 Pawn은 서버가 기존 전장 스폰 위치에 생성하고 복제하며, 해당 팀의 `BoardPlayerController`가 자동으로 소유한다.
6. 지정되지 않았거나 로드에 실패한 기물은 기존 스켈레탈 메시 임시 캐릭터로 안전하게 대체된다.

캐릭터 Blueprint는 `ArenaCombatant`, `ArenaBlue` 또는 `ArenaRed`, `ArenaPiece_*` 태그를 받는다. 태그는 스폰 완료 뒤 지정되므로 팀 판정이나 기물 종류별 초기화가 필요하면 `OnPossessed` 이후나 별도 초기화 이벤트에서 읽는다. 진형 선택은 마와 상의 보드 배치만 바꾸며 캐릭터 클래스나 팀 카메라를 바꾸지 않는다.

### FPS 컨트롤러·조준·이동·체력·공용 스킬 담당자

- GameMode의 Player Controller Class와 `NewPlayerController`의 부모는 `BoardPlayerController`를 유지한다. 장기 RPC, 팀 배정, 보드/로비/전투 카메라가 이 클래스에 있으므로 별도 PlayerController로 교체하지 않는다.
- 이동, 조준, 데미지, 체력, 자가 재생은 위 슬롯에 넣을 Pawn/Character 또는 그 Actor Component에 구현한다.
- 체력이 0이 되면 **서버에서** `JanggiBoard1 -> Report Arena Combatant Defeated(Self)`를 한 번 호출한다. Actor를 서버에서 Destroy하는 방식도 자동 감지하지만, 명시 호출을 기본 경로로 권장한다.
- 전투 결과의 유일한 보드 처리 함수는 계속 `ResolveArenaBattle`이다. 캐릭터 코드가 보드 기물을 직접 Destroy하거나 좌표를 바꾸면 안 된다.
- `BoardPlayerController`는 실제 Pawn을 팀에 맞춰 자동 Possess한다. Pawn Blueprint의 `Possessed`/`BeginPlay`에서 보드 카메라로 강제 전환하는 로직을 추가하지 않는다.
- 공용 스킬과 체력 컴포넌트는 서버 권한으로 수치를 변경하고, 표시용 체력/피격 값만 RepNotify 또는 RPC로 로컬 UI에 전달한다.

### UI·유리창 균열·효과음·BGM 담당자

`NewPlayerController` Blueprint에서 다음 C++ Blueprint Event를 구현한다.

- `On Arena Presentation Started(Local Team)`: 전투 HUD 표시, 전투 BGM 전환, 진입 SFX 시작
- `On Arena Presentation Ended(Local Team)`: 전투 HUD/균열 효과 정리, 보드 BGM 복귀

피격 균열은 Pawn의 체력 컴포넌트가 로컬 소유 플레이어 UI에 전달하고, 장기판 코드는 체력 UI의 세부 구현을 알지 않게 유지한다. 모든 사운드와 위젯은 Dedicated Server가 아니라 로컬 컨트롤러에서만 생성한다.

### 담당자별 최소 통합 확인표

- 미등록 기물로 충돌했을 때 기존 임시 캐릭터·디버그 승리·보드 복귀가 그대로 동작한다.
- 등록 기물로 충돌했을 때 양쪽 Pawn이 한 번씩만 생성되고 올바른 팀 컨트롤러가 소유한다.
- 클라이언트의 공격·스킬 요청을 서버가 검증하고 체력 결과가 양쪽에 동일하게 보인다.
- 패배 보고 뒤 진 캐릭터와 해당 보드 기물만 사라지고 승자 기물은 규칙대로 이동한다.
- 궁(`General`) 패배 뒤 다음 턴으로 넘어가지 않고 `MatchFinished`와 로비 복귀가 유지된다.
- 전투를 여러 번 반복해도 이전 Pawn, HUD, 입력 매핑, BGM이 남지 않는다.

## 성능 최적화

- 기물 선택마다 월드의 모든 Actor를 검색하던 경로를 제거했다.
- 서버는 생성 시점부터 90칸 기물 캐시를 유지하고 이동·제거 때 해당 칸만 갱신한다.
- 복제 복구가 필요할 때도 `BP_Piece_Base` 하위 기물만 검색한다.
- 기물 스케일 동기화는 32개가 확인되는 즉시 반복 타이머를 종료한다.
- PlayerController가 장기판 Actor를 한 번 찾은 뒤 캐시한다.
- 이동 가능 표시는 선택할 때마다 컴포넌트와 재질을 생성·파괴하지 않고, 지금까지 필요했던 최대 수의 마커와 재질 2개를 재사용한다.
- 턴 UI는 0.25초마다 확인하지만 텍스트나 색이 바뀔 때만 Slate 속성을 갱신한다.
- 하드웨어 레이트레이싱 지오메트리와 효과는 런타임에서 비활성화한다.
- 동적 GI와 반사는 SSGI/SSR을 사용하고 Virtual Shadow Map 대신 일반 Shadow Map을 사용한다.
- 텍스처 스트리밍 풀은 1500MB로 설정하고 실제 VRAM 한도를 준수한다.

## 수동 테스트 절차

1. 에디터를 완전히 종료했다가 다시 실행한다. 새 C++ UI DLL을 확실히 읽으려면 에디터 재시작이 필요하다.
2. 먼저 플레이 옵션에서 `Number of Players=1`, `Net Mode=Standalone`으로 실행해 로비가 표시되는지 확인한다.
3. 로비의 `방 만들기`, `빠른 대전`, `새로고침`, 방 이름 입력란이 같은 시각 스타일로 표시되는지 확인한다.
4. 멀티플레이 검사는 플레이 옵션에서 `Number of Players=2`, `Net Mode=Play As Listen Server`를 선택한다. 이 경우 로비는 건너뛰고 양쪽 진형 선택 화면으로 들어가는 것이 정상이다.
5. 양쪽에서 서로 다른 진형을 선택한다.
6. Blue 창 상단에 `내 차례 · 청`, Red 창에 `상대 차례 · 청`이 표시되는지 확인한다.
7. Blue 기물을 클릭해 합법 목적지에 짙은 테두리와 밝은 중심을 가진 옥색 원형 마커가 나타나는지 확인한다.
8. 마커가 없는 칸을 클릭했을 때 기물이 움직이지 않는지 확인한다.
9. 마커가 있는 빈칸을 클릭해 기물이 이동하고 양쪽 화면에서 위치가 같은지 확인한다.
11. 이동 후 Blue 창은 `상대 차례 · 한`, Red 창은 `내 차례 · 한`으로 바뀌는지 확인한다.
12. 상대 기물이 있는 합법 마커를 선택해 보드가 정지하고 아레나로 전환되는지 확인한다.
13. 청/한 캐릭터가 전장 양 끝에 생성되고 두 플레이어가 각각 자기 캐릭터 뒤에서 중앙을 보는지 확인한다.
14. 하단의 `청 승리로 전투 종료` 또는 `한 승리로 전투 종료` 버튼을 누른다.
15. 승자에 따라 보드 기물이 이동/제거되고 임시 캐릭터와 디버그 UI가 사라지는지 확인한다.
16. 양쪽 모두 원래 보드 카메라로 돌아오고 다음 턴이 시작되는지 확인한다.
17. 궁/왕이 패배하는 충돌에서는 다음 턴으로 넘어가지 않고 양쪽 상단에 게임 종료와 승리 팀이 표시되는지 확인한다.
18. 게임 종료 뒤 기물을 클릭해도 선택·이동이 되지 않는지 확인한다.
19. `로비로 돌아가기`를 누른 뒤 양쪽 창 모두 대국실 로비와 같은 오버헤드 카메라로 돌아오는지 확인한다.
20. 화면 좌측 상단에 레이트레이싱 메모리 초과와 VSM 큐 오버플로 경고가 다시 나타나지 않는지 확인한다.
21. 실제 Steam 검사는 패키징된 빌드와 서로 다른 Steam 계정 두 개로 진행한다.
